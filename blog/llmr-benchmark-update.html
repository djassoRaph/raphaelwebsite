Project update

LLMR now has a real home. Here's what the numbers actually look like.

May 7, 2026  ·  ~6 min read  ·  LLMRAIOpen Source

A while back I wrote about LLMR,a JSON site index format that gives AI systems a machine-readable map of your website instead of making them parse every HTML page one by one. The original post was rough. The idea was there, the numbers were there, but the project itself was a single GitHub repo with no real home.

That changed. Here's what's been built, what the benchmark actually shows on a real site, and why this is still worth your attention if you're building anything that touches AI.

What LLMR is, in one sentence

Instead of an AI agent fetching 13 HTML pages to understand your site, it fetches one JSON file,and gets the same information in a fraction of the tokens. Think RSS, but the audience is a language model, not a feed reader.

The benchmark on this site (13 pages, real HTML, no cherry-picking):

Method Bytes Est. tokens HTTP requests
Fetch all 13 pages as HTML 633,997 ~30,000 13
Fetch the LLMR index 12,828 ~3,300 1
Reduction 98% 89% 13× fewer

Those numbers are real. Measured on raphaelreck.com, 13 pages, counted with an actual tokenizer. The LLMR file is 12,828 bytes. The 13 HTML pages total 633,997 bytes. That's not an estimate.

What's changed in the project

When I first posted about this, the project was a concept with a spec draft and a Python script. Since then:

  • open-llmr.org is live,proper site, proper docs, formal v2.0 spec
  • The project moved to a GitLab group to support multiple repos cleanly
  • Generators exist for Python (pip install llmr-generator), npm, WordPress, and Laravel
  • An online generator lets you paste any URL and get a ready-to-deploy llmr.json without installing anything
  • The spec is now formally versioned at v2.0, with compressed keys designed to minimize token usage in the index file itself

The about page at open-llmr.org/about has the full story of why I built it. Short version: I was building an AI agent, watched it burn 30,000 tokens just to understand a 13-page site, and got annoyed enough to do something about it.

How the v2.0 format looks

The key design decision in v2.0 is compressed key names. Every byte in the LLMR file is a token that an AI will consume,so the format itself is designed to be miserly:

{
  "v": "2.0",
  "s": {
    "d": "raphaelreck.com",
    "a": { "n": "Raphaël Reck", "r": "IT_sys_sw_consultant" }
  },
  "p": [
    {
      "id": "llmr-benchmark-update",
      "u": "/blog/llmr-benchmark-update.html",
      "d": "2026-05-07",
      "tg": ["llmr", "ai", "open-source"],
      "rt": 6,
      "sum": "LLMR project update: new site, generator, real benchmark numbers."
    }
  ]
}

v instead of version. u instead of url. tg instead of tags. Looks terse to a human; saves real money at scale when AI agents are querying your site thousands of times a month.

Cost at scale

This is the part that makes the case for anyone running a product with AI features. At Claude Sonnet pricing ($3 per million input tokens):

AI queries / month Without LLMR With LLMR Saved
1,000 $90 $10 $80
10,000 $900 $100 $800
100,000 $9,000 $1,000 $8,000

These aren't my costs. They're the costs that any AI agent,yours, your users', a crawler,pays to understand your site. The bigger the site, the worse the ratio gets without LLMR.

How it relates to llms.txt

You may have heard of llms.txt,a proposed markdown file that gives AI systems a plain-text overview of your site. LLMR is not a replacement for it. They solve different problems:

  • llms.txt is a human-readable overview,good for telling an AI "here's what my site is about"
  • LLMR is a machine-optimized index,good for giving an AI structured, queryable metadata about every page

Serve both. They complement each other. The SEOptimer audit on this site already flagged the missing llms.txt as a recommendation,it's on the list.
